有限状态机是一种非常强大的编程模式。
polystate的核心设计理念：
1. 在类型上记录状态机的状态
2. 通过组合类型实现组合状态机

实际达到的效果是通过类型确定程序的控制流，通过组合类型结构化的使用控制流。当然我更喜欢的说法是组合语义。
这是命令式编程的伟大进步：我们可以通过组合式声明式来确定程序的整体行为，可以通过组合简单状态构建复杂程序。
组合状态的目的是为了组合语义，同时也达到了复用控制流，复用代码的效果。
这种基于语义的复用效果同时达到了：简洁，正确，高效这三重效果。
这个库能够自动成动生成状态图，而状态图就是这个程序的整体行为逻辑。

实际上我们可以看到顺序，分支，循环这三种基础的基础结构都可以通过这种状态机+消息的方式来实现。

允许你制作通用组件，极大提高代码的利用率。

pub fn checkPinST(success: polystate.sdzx(Atm), failed: polystate.sdzx(Atm)) type {
    return union(enum) {
        Successed: polystate.Witness(Atm, GST, null, success),
        Failed: polystate.Witness(Atm, GST, null, failed),
    }
}
如果我们有一个ATM，当我们处于checkPin的状态时我们会要求用户从外部输入Pin，然后检查Pin是否正确，如果正确那么进入到Successed 对应的状态，如果错误进入到Failed的状态。这便是这个状态的全部语义了。而一个常见的要求是：用户最多尝试输入三次Pin，如果三次都错误那么就应该退卡，并返回到开始的界面。这里的三次是一个及其重要的安全参数，不应该被改变。
于是我们可以通过下面的声明来保证程序的整体行为符合我们的预期：
InsertCard: Wit(.{ Atm.checkPin, Atm.session, .{ Atm.checkPin, Atm.session, .{ Atm.checkPin, Atm.session, Atm.ready } } }),
这里展示了组合性的妙用：通过嵌套声明三次checkPin，保证检查Pin的行为最多发生三次。


你可能会好奇这种做法有效果吗？
我这里有一个有趣的例子：两阶段选择，你需要选择一个建筑放置到网格中。我们需要两次选择：第一次选择一个建筑，第二次选择一个位置。 建筑的选择会限制位置的选择。在这个例子中选择已经变成了一个通用的状态，它更准确的定义是：通过鼠标选择。 
最终上面的两阶段选择的语义表达为：
ToPlace      : Wit(.{ Example.select, Example.play, .{ Example.select, Example.play, Example.place } }),
由Example.place提供选择需要的几个具体函数，毕竟place需要两阶段选择的建筑和位置来具体放置建筑，所以Example.place 知道什么是正确的建筑和位置。
这段代码是及其简洁的描述了你的想法，但如果你查看状态图你会发现它的实际状态是很复杂的，因为选择并不简单：选择的具体行为通过三个状态，十个消息以及两个额外的状态参数一同构成。
